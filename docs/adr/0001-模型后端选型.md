# ADR-0001: 模型后端选型 — Embedding 本地 42model + LLM 云端

## 上下文
RAG 系统需要可独立运行的模型后端, 不能依赖特定云厂商的绑定环境(否则离了其 API key 无法运行, 不具备可复现性)。

## 选项
- **A. Ollama + bge-m3 + qwen2.5**: 需下载约 5G, 16GB 内存跑 7B 偏紧
- **B. 全 42model 本地(emb + LLM)**: 零下载, 但 `minicpm5:1b` 在 Metal GPU 上数值异常(连续重复 token 被 42model 自检中止), CPU 全量评估约 87 分钟不可行
- **C. 全云端 API**: 要 key, embedding 也走云端, "本地可复现"打折扣
- **D. Embedding 本地 42model + LLM 云端(混合)**: emb 已验证稳定, LLM 云端快稳

## 决策
选 **D**。

## 理由
- Embedding(`qwen3-embedding:0.6b`) 在 42model GPU 下稳定验证通过(1024 维), 留本地零计费、零下载
- LLM(`minicpm5:1b`) 在 Metal 上数值不稳(已实测), 改用云端 `Qwen2.5-7B-Instruct` 兼顾质量与速度
- 两端均走 OpenAI 兼容接口, 代码用 `langchain-openai` 标准客户端, 切换厂商只改 `.env`

## 后果
- **正面**: embedding 本地零成本, LLM 云端快稳, 配置解耦
- **负面**: LLM 依赖云端 key, "完全本地"打折扣; judge 与 generator 同模型(Qwen2.5-7B)仍有自评偏差
- **缓解**: `.env` 配置化可切换任意 OpenAI 兼容厂商; README 说明该局限; 后续可换更强 judge 模型

## 演变记录
初版计划用 Ollama, 发现本机已装 42model 且自带 embedding+LLM, 改为全 42model; 实测 LLM 在 Metal 数值异常后, 拆分为"emb 本地 + LLM 云端"。两次调整均基于实测, 非预设。
