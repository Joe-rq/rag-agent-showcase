# ADR-0004: 进阶检索选 HyDE

## 上下文
朴素 RAG 用 question 直接检索, 但 question 与文档表达差异大时(短问句 vs 长论述), embedding 相似度不准。

## 选项
- **none**: 朴素检索(基线)
- **hyde**: 假设文档嵌入(先让 LLM 生成假设答案, 用其 embedding 检索)
- **rerank**: 检索 top-N 后用 cross-encoder 重排(需额外 reranker 模型)

## 决策
选 **hyde**。

## 理由
- HyDE 把"问题语义"对齐到"文档语义"空间, 缓解 query-document 表达差异
- 只需复用现有 LLM(DeepSeek), 无需额外 reranker 模型, 部署简单

## 评估验证(20 QA)
实测 HyDE 在本场景**无显著提升**(4 指标 delta ±0.03, context_precision 略降 -0.033)。原因: QA 从 chunks 合成, query ≈ document, HyDE 的"对齐语义"优势无发挥空间(context_recall 两版均 0.97, 朴素检索已命中)。HyDE 真正适用场景是 query 与 document 表达差异大的场合(口语化提问 vs 学术文档、短问句 vs 长论述)。

结论: 进阶检索要看场景匹配, 不是万能的——这本身是有价值的工程判断。后续可叠加 rerank 或改用真实用户提问风格 QA 集再验。详见 evaluation-report。

## 后果
- **正面**: 部署简单, 无额外模型依赖; 在 query-document 差异大的场景有潜力
- **负面**: 每次检索多一次 LLM 调用(延迟 +1-2s); 本场景实测无增益, 若 LLM 假设文档偏离还可能检索错
- **后续**: 叠加 rerank(bge-reranker), 或换真实提问风格 QA 集放大 query-document 差异再验
